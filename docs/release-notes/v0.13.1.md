# javalens-manager v0.13.1 — Linux aarch64 builds + arch-aware install script

Packaging patch on top of v0.13.0. No behavioural changes in the manager itself; this release exists so the `curl … | bash` one-liner works unchanged on aarch64 Linux machines (NVIDIA DGX Spark / GB10, Ampere, Graviton, Raspberry Pi 5 64-bit, etc.).

## What changed

### CI matrix — aarch64 release artifacts

[`.github/workflows/release.yml`](../../.github/workflows/release.yml) gains a second matrix entry on GitHub's free `ubuntu-22.04-arm` runner (Cobalt 100, GA for free public-repo use since 2025-01-16). Every `v*` tag now produces both arches in the same draft release:

- `javalens-manager_<version>_amd64.AppImage` / `_amd64.deb` — x86_64
- `javalens-manager_<version>_aarch64.AppImage` / `_arm64.deb` — aarch64

(The asymmetric `aarch64` vs `arm64` naming is upstream Tauri's choice: AppImages tag with `aarch64`, Debian packages with `arm64`. The install script handles both conventions.)

Both rows stay on **Ubuntu 22.04** so the resulting binaries link against glibc 2.35 and run on any reasonably modern Linux without forcing 24.04's glibc 2.39 baseline on users.

### install.sh — auto-detects host architecture

[`install.sh`](../../install.sh) now reads `uname -m` and selects the matching AppImage from the latest GitHub release:

| `uname -m` | Picks asset matching |
|---|---|
| `x86_64` | `*_amd64.AppImage` |
| `aarch64`, `arm64` | `*_aarch64.AppImage` |
| anything else | exits with an explanatory error |

The same one-liner works on both arches:

```bash
curl -sSL https://raw.githubusercontent.com/hw1964/javalens-manager/main/install.sh | bash
```

If a future release ships only one arch (e.g. an x86-only hotfix), the script fails fast with a message pointing to the releases page rather than installing the wrong binary.

## Compatibility

- Drop-in upgrade from v0.13.0. No data-format changes; existing `projects.json` works as-is.
- The x86_64 build path is byte-identical to v0.13.0's — the only diff in the artifact pipeline is that an additional aarch64 job runs alongside it.
- Existing aarch64 users who installed manually from a self-built v0.13.0 .deb can switch to the published artifact from v0.13.1 onward without any migration.

## Tests

- 42 / 42 manager Rust unit tests still pass (no source changes; the build itself is the test for the new CI path).
- Workflow change verified by tagging v0.13.1 — both arches build and publish in parallel.
