# javalens-manager v0.13.0 — Tray menu refined for AppIndicator + name-after-rename bug fix (paired with fork v1.7.0)

Sprint 13 cutover. The v0.12.0 tray menu shipped colored `IconMenuItem` disks for per-workspace status; empirical testing on GNOME revealed those disks never reach the user — the AppIndicator extension strips per-menu-item images at the D-Bus boundary. v0.13.0 replaces them with monochrome unicode bullets that survive the appindicator pipe, refines the menu shape around what GNOME's tray protocol can actually deliver, and fixes a stale-name-after-rename bug. Paired with [fork v1.7.0](https://github.com/hw1964/javalens-mcp/releases/tag/v1.7.0), which adds 11 new MCP tools across Ring 2 / Ring 3 / Ring 4 (73 tools per service).

## What's new in the manager

### Tray menu — final shape

```
Open dashboard
─────
Workspaces            (header)
  ●  Javalens_WS
  ○  ORB_JATS_WS
─────
Start all services
Stop all services
─────
Quit
```

- **Open dashboard** — raises the main manager window (the dashboard is its default view). Replaces the v0.12.0 "Show" entry; the rename is intentional, "Show" was vague.
- **Workspaces** — disabled header, just groups the per-workspace rows below.
- **Per-workspace rows** — one row per workspace with a monochrome unicode bullet:
  - `●` running
  - `◐` starting / stopping
  - `○` stopped
  - `✗` failed
  Click a row to **toggle** that workspace (stopped/failed → start, running → stop).
- **Start all services / Stop all services** — fan out across every loaded workspace.
- **Quit** — opens the existing quit prompt.

The bullet refreshes within ~1 s of any state change (workspace rename in the dashboard, manual start/stop in the main window, external `kill` of a javalens process). Steady state is silent — the menu is rebuilt + re-broadcast over D-Bus only when the `(workspace_name, phase)` snapshot actually changes since the last tick, so there's no flicker on a stable system.

### Why monochrome instead of colored

Three GNOME / libayatana-appindicator constraints (verified empirically during Sprint 13):

1. **`IconMenuItem` images are stripped** at the AppIndicator D-Bus marshal boundary. The colored 16×16 disks v0.12.0 painted as `Image::new_owned(rgba, …)` never reached the GNOME shell — the indicator extension dropped the per-item icon widgets. The colored circles you may have seen referenced in v0.12.0 docs were invisible in practice.
2. **Left-click on the indicator routes directly to the menu**. `on_tray_icon_event` doesn't fire on this path; `show_menu_on_left_click(false)` is silently ignored. We can't intercept the click to open something else.
3. **A menu is mandatory** — a tray indicator without a menu doesn't register at all on libayatana-appindicator.

Together these make a "rich tray UI" (small + colored + animated) fundamentally impossible without bypassing the system tray entirely. The honest design under the constraints is a polished native menu, which is what v0.13.0 ships. Monochrome unicode bullets render in the menu's own font (1× line height, sized with the surrounding text), so they fit cleanly without dominating each row the way emoji glyphs do.

### Bug fix — tray menu now follows workspace renames

`workspace_status_summary` previously took each workspace's name from the cached `RuntimeStatusRecord` snapshot in `RuntimeManager::snapshots`. That snapshot's `workspace_name` field was set when the runtime was last started or stopped — so renaming a workspace in the dashboard didn't propagate to the tray until the workspace was restarted (because the cache wasn't invalidated on rename).

v0.13.0 changes the loop to read `workspace_name` from the live `config_store.list_projects()` mutex (always fresh, updated atomically by `rename_workspace`). The runtime snapshot is now consulted only for `phase`, where its caching is correct. End result: rename a workspace in the dashboard → tray menu shows the new name within ~1 s.

### Refresh strategy — 1 s poll with change detection

The periodic refresh thread now ticks every second instead of every five. To avoid menu-flicker on every tick, the refresh function caches the last `(workspace_name, phase)` snapshot and skips the rebuild + `tray.set_menu` swap when the new snapshot equals the cached one. So the actual D-Bus traffic to GNOME's appindicator extension only fires on real state transitions; idle CPU stays effectively zero.

## Compatibility

- Drop-in upgrade from v0.12.0. No data-format changes; existing `projects.json` works as-is.
- Tray menu shape changed (new "Open dashboard" naming, monochrome bullets, no more "Workspaces" Unicode-colored emojis). Users who relied on the colored emoji glyphs from a v0.12.0 fallback build will see the new bullets after upgrading.
- New `Open dashboard` text replaces `Show` — same behaviour, clearer name.

## Paired fork release

[Fork v1.7.0](https://github.com/hw1964/javalens-mcp/releases/tag/v1.7.0) ships **11 new MCP tools** picked up automatically by the existing release-poller, taking each workspace service from 62 → **73 tools**:

- **Ring 2 — code generation (6):** `generate_constructor`, `generate_getters_setters`, `generate_equals_hashcode`, `generate_tostring`, `override_methods`, `generate_test_skeleton`.
- **Ring 3 — build & dependency management (3, Maven-only):** `add_dependency`, `update_dependency`, `find_unused_dependencies`. Gradle support is explicitly v1.8.x.
- **Ring 4 — formatter / workflow polish (2):** `format` (file/package/project/workspace scope), `optimize_imports_workspace`.

See the fork's [v1.7.0 release notes](https://github.com/hw1964/javalens-mcp/releases/tag/v1.7.0) for the input/result contract and known limitations (one happy-path test ships `@Disabled` due to the Tycho-surefire fixture-build gap inherited from v1.6.0).

## Tests

- 42 / 42 manager Rust unit tests pass (unchanged from v0.12.0; the rename fix in `workspace_status_summary` reads from a mutex that the existing tests already exercise).
