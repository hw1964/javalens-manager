# javalens-manager v0.10.4 — Named workspaces + transparent live updates

The headline change of Sprint 10: **named workspaces replace per-project ports**, and project mutations propagate to the running javalens process within ~1 s without any agent-session reload, mcp.json rewrite, or subprocess restart. Combined with fork **`v1.4.0`** (required), one javalens process serves an entire workspace's projects to all connected MCP clients simultaneously.

For users at the 28-projects-as-services scale, the practical win is the tool registration count: from **28 services × 66 tools = 1,848** down to **66 per active workspace** (66–198 for typical 1–3 concurrent workspaces). Well under the 100-tool cap once GitKraken's MCP shim is also dropped.

## What changed

### Workspaces, not ports
- Every project belongs to a **workspace** identified by a name (e.g. `jats`, `orb`, `strategies-orb`). Multiple projects sharing a workspace name run as one MCP service backed by one javalens process.
- The legacy `assignedPort` concept is gone. There is no port range setting, no port allocation, no port conflicts — the port was vestigial under stdio anyway.
- The Project Form's port input is replaced by a workspace selector ("New workspace…" or pick an existing one). Existing projects auto-migrate into default workspace names like `workspace-11100` derived from their old assigned port; rename via the **Move…** button on a project row.

### Transparent live updates via `workspace.json`
- The manager writes a `workspace.json` file to each workspace's Eclipse data dir at `~/.cache/javalens-manager/workspaces/<workspace-name>/workspace.json`. It lists the workspace's project paths.
- Fork v1.4.0's `WorkspaceFileWatcher` picks up changes within ~1 s (`java.nio.file.WatchService`, event-driven) and calls `addProject` / `removeProject` on the live `JdtServiceImpl`. The agent's next tool call sees the updated workspace.
- Adding or removing a project within a workspace **never** changes mcp.json. mcp.json only changes when a workspace is created or destroyed.
- Atomic file write (temp + rename) so the watcher never observes a half-written file.

### Per-workspace MCP service ID
- mcp.json entries are keyed by workspace name: `jl-jats`, `jl-orb`, etc. — one entry per workspace.
- The mcp.json `command` runs `java -jar javalens.jar -data <workspace-data-dir>` (no project paths in args; javalens reads them from `workspace.json`).
- Cursor / Claude / Antigravity / IntelliJ each spawn their own javalens subprocess per workspace, but every spawn loads from the same `workspace.json` so they all see the same set of projects.

### Workspace lifecycle
- **Start a project** — manager writes (or updates) `workspace.json`, then either spawns a new javalens for the workspace or registers the project as a member of an already-running workspace process. The agent sees the workspace's full project set on the next tool call.
- **Stop a project** — the project leaves its workspace. The workspace process keeps running for any remaining members; only when the last member leaves is the process killed.
- **Stop a workspace** (force-stop) — kills the entire workspace process unconditionally regardless of how many members are attached.
- **Move a project to another workspace** — manager rewrites both the source and destination `workspace.json` files; running javalens processes pick up the change.
- **Delete a workspace** — stops the process, removes every member project record, removes the JDT data dir.

### New Tauri commands
- `set_project_workspace(projectId, workspaceName)` — move a project between workspaces. Replaces the legacy `update_project_port`.
- `rename_workspace(oldName, newName)` — bulk-update every member project's workspace_name. The new MCP service ID is emitted on the next deploy.
- `delete_workspace(workspaceName)` — kills the workspace's javalens process, deletes every member project record, removes the JDT data dir.

### What's preserved
- Existing MCP-deploy targets (Cursor, Claude, Antigravity, IntelliJ) work as before. The mcp.json schema is unchanged — only the *number* of entries changes (one per workspace instead of one per project).
- Existing release-source plumbing (switching between `hw1964/javalens-mcp`, upstream Pzalutski, or a custom repo) is unchanged.
- Existing Workspace Import (`.code-workspace` discovery) works; pick the target workspace before clicking **Import selected**.

## Migration (automatic on first launch)

When you launch v0.10.4 with a v0.10.3 `projects.json`:
1. Each project's `assignedPort` becomes the suffix of a default workspace name: `workspace-11100`, `workspace-11101`, etc.
2. The manager writes `workspace.json` files into the JDT data dirs for each workspace.
3. The migrated `projects.json` is rewritten in the new format. Subsequent reads are clean.

You can rename `workspace-NNNN` to anything more meaningful via the **Move…** button on a project row, or by editing `projects.json` directly.

## Requires

- Fork **`hw1964/javalens-mcp` v1.4.0** for the `WorkspaceFileWatcher`. The default `releaseRepo` setting points at the fork; if you switched to upstream Pzalutski's javalens it will not have the file watcher and live updates will not work — switch the release source back to the default in Settings.

## Tests

**37 / 37 unit + integration tests passing.**

| Category | Count | Coverage |
|---|---|---|
| Pre-existing | 11 | release_manager, command_spec, slugify, bootstrap, etc. |
| Workspace pure-function tests | 15 | `sanitize_workspace_name`, `ConfigStore` workspace methods, `mcp_server_id_for_workspace`, `read_projects` migration |
| `workspace.json` I/O | 5 | atomic write, empty-paths-removes-file, mkdir, overwrite, no-op |
| `RuntimeManager` lifecycle | 6 | grouped spawn (join), keep-alive on partial leave, kill on last leave, force-stop, `remove_project_runtime`, distinct workspaces independent |

Internal refactors made testable:
- `write_workspace_json_to_dir` extracted as a `pub(crate)` free function so I/O can be tested without the full ManagerService dependency graph.
- `RuntimeManager::start_runtime_with_spec` extracted from `start_runtime` so lifecycle tests can spawn a stand-in `sleep 60` instead of `java -jar javalens.jar`.

## Known limitations / follow-ups

- **Grouped Dashboard view** (one card per workspace with project rows nested) is deferred — the flat list still works and shows the workspace per row.
- **Inline workspace rename / delete UI** in the Dashboard is not yet wired; the Tauri commands exist (`rename_workspace`, `delete_workspace`) but only "Move project" has a UI affordance (a `prompt()` from the project row).
- **Sprint 11** will close the detection-matrix gaps (Tycho-aware Maven, workspace bundle pool for `Require-Bundle`, Gradle Tooling API) and ship the structural refactoring tools (`move_class`, `pull_up`, `push_down`, `encapsulate_field`). See `docs/sprint-11-backlog.md`.

## Install

Download the `.AppImage` or `.deb` asset for your distribution. The first launch auto-migrates any existing `projects.json` into the new schema.
